PASCAL Technical Memo 5.1

Documentation of Internal Structure


     The primary documentation of the compiler internals is the code itself. However, as the listings for each phase (called Pass1, Pass2, Pass3, and PMD) are long, we have supplied in this memo a prose description of the major structures, both procedure and data, of each phase. It is intended that the reader should have digested Tech. Memos 1.1, 2.1, and 3.1 prior to starting on this one, so that he is familiar with the overall structure of the compiler, the distribution of function among phases, and the internal forms of the program and semantics tables in interpass communication.

     After reading the description of a phase in this memo, then with the aid of a cross reference listing of identifiers (produced by the utility program XPLXREF) and of a compilation listing (produced by XCOM4) one should be able to begin selective study of the source code.

Caveat: This documentation is based on Release1, Version 1 of the compiler (June 1, 1976) and it is unlikely that the documentation will ever be revised to keep abreast of updates to the compiler itself.

1.  Pass1 internal description

     Pass 1 is organized as five principal modules embedded in a control program environment. In this description we shall give the function associated with each module, describe principal aspects of the internal data base of each, and describe the input-output files and inter-module communications protocols. For more detail, one can refer to a listing of the source code and a cross-reference listing (use the XPL compiler and the utility program XPLXREF to obtain these).

1.1  Lexical analyzer is responsible for getting source program text from the SYSIN file, analyzing the source text to produce syntactic token codes, identifier or literal constant names, and conversion of numeric constants to their internal representation as 32-bit, two's complement, binary radix integers or as single-precision hexadecimal floating point numbers.

The principal procedures are:

     CHAR
     GET_CARD
     SCAN

and

     FLOAT
     REAL_ADD
     REAL_MULTIPLY
     TENINTO

The latter group performs real number conversions and is called from SCAN.

     GET_CARD performs card reading and listing. It returns the image of an input record in a global, CHARACTER type variable, TEXT. When the end of the SYSIN file is detected, TEXT is set to a string of end-of-file symbols. The ordinal number of the currently active input record is kept in the global variable CARD_COUNT.

     CHAR is a procedure used to simulate character stream input; each call leaves the next character under the text string index pointer.

     SCAN is the principal working procedure of the lexical analyzer. It analyzes the string in TEXT, discards portions as they are analyzed, and generates a call to GET_CARD upon reaching a record boundary. The currently scanned character is indexed within TEXT by the index variable CP. (Index values start from zero, in the XPL implementation languages.) Analysis involves a case switch on the equivalence class of the leading character of the string to be analyzed. Sequences of blanks, comments, quoted string constants, and identifiers are scanned rapidly by employing the TRT (translate and test) instruction. Other sequences are scanned character-by-character. Reserved words are identified by searching a table, indexed by word length. Toggles appearing within comments are detected and switched by SCAN.

The data base for lexical analysis includes

     V -

an array of character string identifiers of lexical token types sorted as terminal, nonterminal tokens and secondarily by length.

     V_INDEX -

indexes the terminal token names of V by length.

     RESERVED_LIMIT -

a constant; the length of the longest reserved word.

     MARGIN_CHOP -

record margin boundary.

     CHARTYPE, TX, NOT_LETTER_OR_DIGIT, ALPHATABLE, BLANKETABLE,
     COMMENTABLE, STRINGTABLE -

character translation tables.

     BUFFER, TEXT -

character strings to hold input record and portion being analyzed.

     TEXT_LIMIT, CP -

index pointers into text.


     Inter-module communication variables set by SCAN are:

     TOKEN -

code number of lexical token scanned;

     BCD -

character string value of token scanned;

     NUMBER_VALUE -

numeric conversion of token scanned;

     CONTROL -

an array of the values of compiler control toggles set when '$' is encountered in scanning comments.

ordinal number of the current input record.


1.2  Diagnostic listing editor (DGNS)

     This module prepares an edited listing of the source program and sends it to a sequential file for later use by the execution profile routine, following execution of the program. The diagnostic listing is prepared only if the compiler parm DEBUG is set to a value of 2 or greater. This value is set by the monitor in the predeclared global variable MONITOR_LINK(2). The procedure of the diagnostic listing module is

     PARAGRAPH

which is called from the syntactic module each time that a new syntactic token is obtained. Subordinate procedures and internal variables of this module have DGNS# as a prefix to their identifiers.

     Global variables referred to by this module include

     BCD
     TEXT
     TOKEN

It produces no intended side effect, apart from writing the file FILE3.

1.3  Lexicon and semantic table module

     The compiler maintains a lexicon in the form of a hash-indexed array of internal, sequential files. Each lexical declaration block (procedure, function or record block) is given a unique block number, and this number is incorporated as part of the hash code. Since each hash bucket defines a unique sequential file, there is no way that identifiers of the same name, but declared in different blocks, can ever appear in a common file. Identifier lookup within a sequential file is by sequential search. Nested contextual blocks are accounted for by maintaining a stack of the block numbers of currently open blocks. The block number of a block is saved in the VALUE field of the semantic table entry for the block heading.

     Although most of the semantic table is built in Pass 2, some semantic data are recorded in Pass 1. Semantic table entries are allocated for identifiers and constants. The field of the semantic tables are kept in fixed-dimension arrays with a common index, as this is the only data structuring tool available in the XPL implementation language. These arrays are called;

     IDENTITY -

holds the identifier strings.

     DATATYPE -

holds an index pointer to an entry for the datatype of a variable or function.

     VAR_TYPE -

holds a type code that designates the category of an object, i.e., constant, variable, procedure, etc.

     VALUE -

holds the internal representation of numeric constants, or the block number associated with procedures, functions and records.

     WHERE_DECL -

holds the input line number of the first appearance of an identifier.

     SYMB_LINK -

holds the index pointer to the next item in an internal, sequential file headed by a hash-code bucket.

     N_LOOKUPS -

holds a count of the number of subsequent references to an identifier.

1.4  Syntactic analysis and error recovery

     The Pascal/360 compiler employs top-down, predictive parsing by a table-driven LL(1) parser. Error recovery is also predominately table-driven, and employs a multi-level strategy that keys on reserved words and punctuation symbols to re-establish a legal syntactic context. The principal procedure of the parser is:

     COMPILATION_LOOP

     COMPILATION_LOOP uses as a data base several tables prepared by an LL(1) constructor. These tables determine which production to apply, if any is applicable, for each pair (N,t) where N is a nonterminal symbol of the LL(1) grammar describing Pascal, and t is a terminal symbol. The tables implement a sparse representation of the decision matrix, and are therefore not easily read. However, the parsing decision tables occupy only 2078 bytes of storage, although there are 62 terminal and 96 nonterminal symbols of the grammar. Figure 2 illustrates the data structure that implements the decision matrix. GET_TOKEN is a subsidiary procedure of COMPILATION_LOOP from which calls to the lexical analyzer and the symbol table lookup routines are made. GET_TOKEN also performs the semantic action of constructing a leaf node of the syntax tree for each <identifier> or <constant> token. However, this action is not taken at the time that the token is scanned. Semantic action is not taken on a token before it has been processed by COMPILATION_LOOP.

THE SYNTACTIC ERROR RECOVERY STRATEGY


     In an LL(1) parser, a stack contains a list of the syntactic tokens that are predicted to complete a sentential form from the point of scanning to its end. The tokens in the stack may be either terminal or nonterminal, and nonterminal tokens may have empty productions, allowing the prediction of optionally occurring phrases. For each nonterminal token, the parsing tables contain, a list of the terminal tokens that could legally appear at the head of the unanalyzed portion of the input string, and for each valid (nonterminal, terminal) pair, gives the production to apply. When a production is applied its right hand side is stacked in reverse order, replacing the nonterminal that is the left side symbol of the production.

A syntax error is detected when either

1)   the symbol at the top of the stack is a terminal, and it is not matched by the head of the input string, or


2)   the symbol at the top of the stack is nonterminal, but when paired with the head of the input string, no production is found to apply.


When an error is detected, a call is first made to procedure DIAGNOSE ERROR, whose task is to assemble a list of the terminal symbols that might have been accepted as head of the input string, given the predicted token at the top of the parsing stack. Next, a call is made to SOFT_RECOVER, which applies a sequence of recovery strategies, employing the first one that succeeds, if any does.

     Initially, SOFT_RECOVER tries a language-dependent heuristic. Certain punctuation symbols are fairly commonly substituted for one another, either because of their proximity on a keyboard, such as , and . or as with = and :=, because = is used for the assignment operation in other commonly used program languages. If a substitute symbol would be acceptable in place of the one actually found at the head of the input string, then the substitution is made and the parse continues.

     If no authorized substitution works, and the head of the input string is an identifier, whereas a keyword would have been expected, then the strategy is to check to see whether the identifier differs from an acceptable keyword by no more than a single letter insertion, deletion or substitution. If it does, then a spelling error is presumed, and the keyword is substituted.

     If these attempts to recognize a trivial error of typing or spelling fail, then SOFT_RECOVER tries the hypothesis that a terminal symbol has been omitted from the input string. It looks at the next symbol in the parsing stack, beneath the stacktop symbol, and asks whether the head of the input string could be accepted from that predicted symbol. If it could, then the stacktop symbol is popped off, and the parse continues. This strategy accounts for missing 'END' symbols, among others. Even if this attempt fails, the missing token hypothesis is not abandoned. If a unique terminal symbol was expected, then it is supplied and the parse continues. If a semicolon or a right parenthesis could have been expected, then one or the other of these will be supplied. Only a single token will be gratuitously supplied, however, before additional parsing of the input string has taken place. The insertion of an extra semicolon before an 'ELSE', a habit of PL/1 programmers, is treated as a special case, to be noted and corrected. If these strategies do not produce local recover, then a more severe strategy is pursued. SCAN_AHEAD advances the token scanner on the input string, looking for a symbol designated as a 'catch' token, on which local recovery is likely to be possible. After a final attempt to recover on a catch symbol from the existing stack configuration, HARD_RECOVER is called to reconfigure the parsing stack.


HARD_RECOVER pops the parsing stack until one of four things happens:

1)   A token appears at the top of the stack such that recovery on the catch symbol is assured, and the parse continues from that point;


2)   A nonterminal token appears at the top of the stack such that the catch symbol could be an element of a phrase produced from the token. In such a case, the stack is reconfigured by replacing the stacktop token by the tail of a phrase that could be produced from it, and which predicts the catch token at the head of the input string.


3)   A token is encountered at the stacktop which could have been preceded in the stack by a token for which condition (2) would have been satisfied. In this case the stack is reconfigured as in (2) except that the stacktop symbol remains embedded in the stack, rather than being replaced.


4)   The parsing stack is emptied. Procedure CRASH_RESTORE reconfigures the parsing stack with a sequence of dummy nonterminal tokens from which recovery can take place on any of a designated subset of the catch tokens. The scanner is advanced in the input string until one of these tokens is encountered.


     This graded sequence of recovery actions assures that at least one production will be applicable following a call to the error recovery module, and that recovery can take place even if context has been lost, as occasionally happens due to a scrambled source program file, or a missing string quote error.

     The semantic module utilizes a large data base including many structures. Its input data include the index of a production to be applied, which is paired as its parameter, and the global variable NODE_NUMBER indicating the code of an internal node of the syntax tree that is to be constructed. Ordinarily, the value of NODE_NUMBER has a fixed relationship to the production number, and is set from the constant-valued translation table NODE#TRANSLATE just before SYNTHESIZE is called. During error recovery, however, NODE_NUMBER may be initialized with a special value to designate an error node in the syntax tree, in order to warn Pass 2 that it is receiving invalid data.

     Syntax trees are grown bottom-up, in spite of the fact that a top-down parsing algorithm is employed. Figure 4 indicates a subsidiary stack, embedded in an array called SUBTREE_STACK, that holds pointers (indices into the array PARSE_TREE) to the roots of a forest of partially grown trees. Another stack, NODE_STACK, holds the syntax code appropriate to an interior node of the syntax tree, and a second column of the same stack, SUBTREE_INDEX, holds a marker for SUBTREE_STACK to designate the first of a string of subtrees for that interior node. Whenever the parse predicts a phrase for which an interior node of the syntax tree is required, a procedure STACK_NODE is called to push the appropriate value onto NODE_STACK. Whenever a symbol table lookup occurs, the procedure GET_TOKEN builds a leaf node of the syntax tree and stacks a pointer to it on SUBTREE_STACK. Whenever a production is applied for which a phrase is completed, the procedure UNSTACK is called. It forms a new node of PARSE_TREE, taking as its code the value popped from NODE_STACK, and as its successors the nodes pointed to by SUBTREE_STACK starting at SUBTREE_INDEX and going up to SUBTREE_STACKSIZE. The newly formed node is then the root of a subtree, and a pointer to it is stacked on SUBTREE_STACK.

     Building syntax trees is the more straightforward of the tasks of SYNTHESIZE. Managing the scopes of declarations is somewhat more complicated, and a large number of status flag variables are employed for this purpose. The semantic tables must contain enough information that when a record variable is referred to, the declaration block containing its field identifiers can be reopened. This information consists of a reference (in the DATATYPE field of the semantic table) to the entry for the record type. The record type contains in the VALUE field to access its field identifiers. Pointer variables must also have their DATATYPE fields filled in, for they may refer to variables of a record type. The same mechanism is employed to associate the identifier of a forward-declared procedure with the main part of its declaration block, which appears later. The procedure ASSOCIATE_TYPE_WITH_ID enters the requisite data into fields of the semantic table.

     The procedure SYNTHESIZE, in its cases for the WITH statement and for variables, manages the setting of the flags and counters to indicate when qualified variable scopes are to be opened and closed. The procedure UNSTACK_QUALIFIER tests the stacked node number when a qualified variable is being analyzed, to determine whether or not to reopen a declaration block.
